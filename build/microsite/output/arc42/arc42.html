1. Introduction and Goals

This document describes the software architecture of the Automotive DDS OTA Update Coordination Demo — a simulation of a multi-ECU Over-the-Air (OTA) firmware update system orchestrated via DDS (Data Distribution Service) pub/sub middleware.

The demo was built to show why DDS beats CAN/LIN for scalable, real-time automotive Software-Defined Vehicle (SDV) communication, specifically in the context of AUTOSAR Adaptive Platform (UCM – Update and Configuration Management).

Table 1. General functionalities of the system
Functionality Description

OTA Update Coordination

The UpdateManager broadcasts a START_UPDATE command via DDS multicast; all ECUs in the DDS domain receive it simultaneously.

ECU State Machine

Each ECU transitions through a 7-state OTA lifecycle: IDLE → DOWNLOADING → VERIFYING → INSTALLING → REBOOTING → DONE (or ERROR).

Metrics Collection

End-to-end latency, per-ECU jitter, throughput, and bus-load are measured and compared against a CAN 2.0B model.

QoS Demonstration

Shows the impact of RELIABLE vs BEST_EFFORT and TRANSIENT_LOCAL vs VOLATILE DDS QoS policies.

Visualization

Five publication-quality charts are generated from demo run data (scalability, latency, throughput, state timeline, QoS impact).

CAN vs DDS Comparison

A built-in CANBusAnalyzer models CAN 2.0B overhead and compares it with DDS metrics side by side.

1.1. Requirements Overview

1.1.1. Functional Requirements

ID Requirement

FR1

The system shall simulate a fleet of N automotive ECUs undergoing a coordinated OTA firmware update.

FR2

The UpdateManager shall broadcast OTA commands to all ECUs using DDS multicast (one write, N recipients).

FR3

Each ECU shall autonomously execute its OTA state machine and publish state transitions on the ECUStatus DDS topic.

FR4

The system shall support configurable QoS policies (RELIABLE/BEST_EFFORT, TRANSIENT_LOCAL/VOLATILE).

FR5

The system shall measure and report per-ECU latency, jitter, throughput, and bus load.

FR6

The system shall compare DDS performance metrics against a CAN 2.0B model.

FR7

A late-joining ECU shall automatically receive current OTA state via TRANSIENT_LOCAL durability — without manual re-publish.

FR8

The system shall generate scalability data for 5, 10, and 20 ECU scenarios.

1.1.2. Quality / Non-Functional Requirements

ID Requirement

QR1

OTA command delivery latency (network) shall be ≤ 5 ms for up to 20 ECUs under DDS multicast.

QR2

CAN bus load shall not exceed 60% at 20 ECUs to remain in the comparison model’s valid range.

QR3

The simulated DDS network delivery shall use a Gaussian latency model (mean 2 ms, stddev 0.5 ms).

QR4

BEST_EFFORT QoS shall simulate ~3% message drop to illustrate reliability trade-offs.

QR5

The ECU fault injection probability (_INSTALL_ERROR_PROB) shall default to 5% for resilience demo.

QR6

The system shall run on Python 3.8+ with no external DDS middleware dependency.

1.2. Quality Goals

The top quality goals for this demonstration system, in priority order:

Priority Quality Goal Motivation

1

2. Architecture Constraints

2.1. Technical Constraints

ID Constraint

TC1

The system must be implemented in Python 3.8+ with no external DDS middleware (no Connext DDS, no OpenDDS, no CycloneDDS). All DDS behaviors are simulated in pure Python.

TC2

The system must run on Linux (Ubuntu 20.04+) or macOS without special drivers or hardware.

TC3

All DDS pub/sub, QoS, and RTPS discovery behaviors are simulated within the dds_abstraction.py module. The simulation must remain faithful to OMG DDS 1.4 semantics.

TC4

Network transport is simulated using a Gaussian latency model (mean 2 ms, stddev 0.5 ms) and in-process message queues. No real UDP/multicast sockets are used.

TC5

The ECU state machine must mirror the AUTOSAR UCM PackageManagement states: IDLE → DOWNLOADING → VERIFYING → INSTALLING → REBOOTING → DONE / ERROR.

TC6

Python package dependencies are limited to those listed in requirements.txt (numpy, pandas, matplotlib, scipy, colorama).

2.2. Organizational Constraints

ID Constraint

OC1

The project is a demonstration / educational artifact — not a production system. MIT License applies.

OC2

The demo must be executable in < 20 seconds for a 5-ECU scenario to be suitable for live presentation.

OC3

Pre-recorded sample data must be available so visualization can run without a live demo execution (suitable for screen recording / slides).

The system is a single-process Python application; all participants share in-process message queues.

Table 4. Technical context

4. Solution Strategy

The fundamental decisions that shape the system architecture:

ID Decision

SS1

Pure-Python DDS simulation — no external DDS library. This makes the demo fully self-contained and portable. The dds_abstraction.py module provides a faithful DDS API (create_participant, create_topic, create_writer, create_reader, write, shutdown) whose behavior mirrors OMG DDS 1.4.

SS2

Pub/Sub with QoS as first-class concept — the architecture models DDS QoS policies (RELIABLE/BEST_EFFORT, TRANSIENT_LOCAL/VOLATILE, Deadline, Liveliness, Latency Budget) as runtime-configurable parameters, not hard-coded behavior.

SS3

Thread-per-ECU concurrency model — each ECU runs its state machine in a dedicated Python thread, allowing realistic parallel state transitions and demonstrating true multicast delivery (single write → N concurrent transitions).

SS4

Dual-mode operation — the system supports both a live demo run (run_demo.py) and a pre-recorded mode (generate_sample_data.py + visualize_results.py), decoupling presentation from live execution risk.

SS5

AUTOSAR UCM state machine alignment — the ECU state machine (IDLE → DOWNLOADING → VERIFYING → INSTALLING → REBOOTING → DONE/ERROR) directly mirrors the AUTOSAR Adaptive UCM PackageManagement state machine, making the demo directly relevant to AUTOSAR Adaptive Platform architects.

SS6

Metrics as first-class outputs — MetricsCollector and CANBusAnalyzer are separate modules that record, analyze, and export data independently from the demo orchestration, enabling offline comparison and chart generation.

5.2. Level 2 — Module Detail

5.2.1. dds_abstraction.py — DDS Simulation Layer

Purpose: Single-module replacement for a real DDS middleware stack. Simulates OMG DDS 1.4 pub/sub, QoS, and RTPS discovery within a single Python process.

Key exported types and functions:

Export Description

ReliabilityKind

Enum: RELIABLE, BEST_EFFORT

DurabilityKind

Enum: TRANSIENT_LOCAL, VOLATILE

QoSProfile

Dataclass: reliability, durability, history_depth, deadline_ms, latency_budget_ms, name

RELIABLE_QOS

Pre-built QoS: RELIABLE + TRANSIENT_LOCAL, depth=10, deadline=100ms

BEST_EFFORT_QOS

Pre-built QoS: BEST_EFFORT + VOLATILE, depth=1

CONTROL_QOS

Pre-built QoS for OTA commands: RELIABLE + TRANSIENT_LOCAL, deadline=50ms

create_participant(domain_id)

Registers a DDS participant in the simulated domain.

create_topic(name, type_name, qos)

Registers a DDS topic in the participant registry.

create_writer(topic)

Returns a writer handle for the given topic.

create_reader(topic, callback)

Registers a callback invoked on each incoming sample; simulates on_data_available.

write(writer, data)

Publishes a sample; applies simulated latency and BEST_EFFORT drop; dispatches to all matching readers.

shutdown(participant)

Deregisters participant; triggers liveliness loss notifications.

Simulated QoS behaviors:

  • RELIABLE: All samples are delivered (no drop). ACK/NACK implied.

  • BEST_EFFORT: ~3% random drop applied at write() time.

  • TRANSIENT_LOCAL: Writer caches last history_depth samples; new readers receive the cache on discovery.

  • Deadline: Tracked per-writer; on_deadline_missed callback fires if write() is not called within deadline_ms.

  • Liveliness: Participants must assert liveness periodically; on_liveliness_changed fires on shutdown().

5.2.2. ecu.py — Simulated ECU

Purpose: Models a single automotive ECU as a DDS participant. Runs its OTA state machine in a dedicated thread.

State machine:

IDLE ──► DOWNLOADING ──► VERIFYING ──► INSTALLING ──► REBOOTING ──► DONE
                                             │
                                             └──► ERROR (5% probability)

DDS role:

  • Subscriber on OTAControl topic (RELIABLE + TRANSIENT_LOCAL) — receives START_UPDATE commands.

  • Publisher on ECUStatus topic (RELIABLE + TRANSIENT_LOCAL) — publishes ECUStateUpdate samples at each state transition and during download progress (10 progress updates per stage).

Key parameters:

Parameter Value

_INSTALL_ERROR_PROB

0.05 (5% installation failure rate, configurable)

ECUStatus deadline

100 ms

OTAControl deadline

50 ms

Liveliness lease

500 ms

5.2.3. update_manager.py — UCM Master / OTA Coordinator

Purpose: Orchestrates the OTA update. Broadcasts the START_UPDATE DDS sample via multicast and monitors all ECUs until they reach DONE or ERROR.

DDS role:

  • Publisher on OTAControl (CONTROL_QOS) — single write() reaches all N ECUs simultaneously.

  • Subscriber on ECUStatus (RELIABLE_QOS) — receives state updates from every ECU; updates real-time ANSI status table.

Key behaviors:

  • Displays a live ANSI-colored status table (ECU ID, state, progress %, latency, last update timestamp).

  • Records per-ECU first-ACK latency (time from broadcast to first ECUStatus sample received).

  • Detects ERROR state and reports resilience metrics.

5.2.4. metrics_collector.py — Metrics & Analysis Engine

Purpose: Records all state transitions, computes performance metrics, models CAN 2.0B overhead, and exports data.

Sub-components:

Class/Function Description

MetricsCollector

Records state transitions; computes per-ECU latency, jitter, throughput, end-to-end latency.

CANBusAnalyzer

Models CAN 2.0B: frame count per update (ceil(payload/8)), bus load %, congestion at N ECUs.

generate_scalability_data()

Produces a DataFrame for 5/10/20 ECU scenarios with DDS and CAN metrics side-by-side.

save_run_results()

Exports per-run metrics to data/run_TIMESTAMP.csv.

  ├─► python run_demo.py --num-ecus 5 --show-comparison

6.3. Error Recovery Scenario

ECU installation error (5% probability)
ECU_003
  │
  ├─ INSTALLING
  │    └─ random() < 0.05 → ERROR
  ├─ publish ECUStatus(state=ERROR, ecu_id=ECU_003)
  │
UpdateManager
  │
  ├─ Receives ECUStatus(ERROR) from ECU_003
  ├─ Marks ECU_003 as FAILED in status table
  ├─ Reports error count in metrics summary
  └─ Continues monitoring remaining ECUs

The demo continues for all other ECUs; the error is reported in the final metrics summary. DDS Deadline QoS on ECUStatus (100 ms) would also trigger on_deadline_missed if ECU_003 silently crashes (vs CAN which requires ~1-3 s timeout detection).

6.4. Late Joiner Scenario (TRANSIENT_LOCAL)

New ECU joining after START_UPDATE broadcast
T=0   UpdateManager broadcasts START_UPDATE via OTAControl
T=5s  ECU_006 joins (late joiner)
        └─ create_reader(OTAControl) triggers TRANSIENT_LOCAL replay
        └─ ECU_006 immediately receives cached START_UPDATE sample
        └─ ECU_006 begins DOWNLOADING without any manual re-publish

This is impossible with CAN: CAN has no durable delivery; a late-joining ECU must poll or wait for the next periodic re-broadcast.

6.5. Processes / Threading Model

Thread Description

Main thread

run_demo.py orchestrator; runs the display loop.

UpdateManager thread

Monitors ECU status queue; updates ANSI table; detects completion.

ECU_001..N threads

One thread per ECU; executes state machine transitions with simulated delays.

DDS delivery thread

dds_abstraction internal dispatcher; applies simulated latency and routes samples to reader callbacks.

8.4. CAN 2.0B vs DDS Comparison Model

The CANBusAnalyzer in metrics_collector.py is a cross-cutting analysis component that overlays CAN 2.0B overhead onto every DDS metrics run.

Table 9. CAN 2.0B model parameters
Parameter Value

Bus speed

1 Mbps

Max payload per frame

8 bytes

OTA state update payload

50 bytes → 7 CAN frames per update

Delivery model

Unicast × N ECUs (O(N) transmissions)

Bus load formula

(frames_per_update × N_ECUs × bits_per_frame) / (bus_speed × period)

Key comparison points demonstrated:

  • 5 ECUs: CAN ~25 ms delivery; DDS ~4.5 ms

  • 20 ECUs: CAN bus load ~60% (congested); DDS <0.01%

  • Scalability: CAN O(N²) bus load growth; DDS O(1)

8.5. Fault Injection

The _INSTALL_ERROR_PROB = 0.05 constant in ecu.py is a cross-cutting failure injection mechanism. When an ECU reaches the INSTALLING state, there is a 5% probability it transitions to ERROR instead of REBOOTING. This demonstrates:

  • DDS RELIABLE QoS does not prevent application-level errors (only network-level loss).

  • MetricsCollector captures and reports error counts alongside normal metrics.

  • The UCM Master detects errors via ECUStatus topic subscriptions and reports them in the summary.

9. Architecture Decisions

9.1. ADR-001: Pure-Python DDS simulation over real DDS middleware

Status: Accepted

Context: The demo needs to run on any developer laptop without installing DDS middleware (Connext DDS, OpenDDS, CycloneDDS) which requires C++ libraries, licenses, or complex configuration.

Decision: Implement a faithful DDS API simulation in pure Python (dds_abstraction.py).

Consequences: * () Zero external dependencies; `pip install -r requirements.txt` is sufficient. * () Full control over simulated latency, drop rates, and QoS behaviors. * (+) Works on Linux and macOS without root/admin privileges. * (-) Not a real RTPS implementation; cannot interoperate with real DDS participants. * (-) Network latency is simulated, not measured from real hardware.

9.2. ADR-002: Thread-per-ECU concurrency model

Status: Accepted

Context: ECUs must transition through states independently and in parallel to demonstrate true multicast delivery (vs CAN’s sequential per-node delivery).

Decision: Each ECU runs its state machine in a dedicated Python threading.Thread.

Consequences: * () Realistic parallel state transitions; UpdateManager sees overlapping ECU progress. * () Demonstrates that one DDS write() triggers N concurrent processing threads. * (-) Python GIL limits true CPU parallelism; for I/O-bound simulation this is acceptable. * (-) Thread count scales linearly with ECU count; not suitable for 1000+ ECU simulation.

9.3. ADR-003: Dual-mode operation (live + pre-recorded)

Status: Accepted

Context: Live demos can fail; presentation environments may not support long-running processes. Charts must be available for slide decks.

Decision: Separate generate_sample_data.py produces realistic pre-recorded CSV files; visualize_results.py works from either live or pre-recorded data.

Consequences: * () Presenter can use either live execution or pre-generated charts. * () Charts are deterministic and reproducible for slide decks. * (-) Pre-recorded data does not reflect real-time variability.

9.4. ADR-004: AUTOSAR UCM state machine alignment

Status: Accepted

Context: The demo targets AUTOSAR Adaptive Platform architects and RTI Connext users. The OTA flow must be immediately recognizable.

Decision: Map the ECU state machine directly to AUTOSAR UCM PackageManagement states: IDLE → DOWNLOADING → VERIFYING → INSTALLING → REBOOTING → DONE / ERROR.

Consequences: * () Demo is immediately credible to AUTOSAR engineers. * () Clearly shows how ara::com DDS binding maps to real OTA workflows. * (-) The simulation simplifies timing (no real firmware binary transfer).

9.5. ADR-005: Single DDS domain for all participants

Status: Accepted

Context: Simplicity of demo; multi-domain would require domain bridge configuration.

Decision: All participants (UpdateManager + all ECUs) share domain_id=0.

Consequences: * () Automatic RTPS discovery finds all participants without configuration. * () Multicast delivery works out of the box. * (-) No isolation between vehicle bus and diagnostics domain (noted in README as extension point).

11. Risks and Technical Debts

11.1. Risks

Table 11. Technical risks
ID Risk Mitigation

R1

Python GIL limits thread parallelism — for large ECU counts (>50), thread scheduling overhead may distort simulated latency measurements.

Acceptable for demo purposes (≤ 20 ECUs). For high-fidelity simulation, use asyncio or multiprocessing.

R2

Simulated latency diverges from real DDS — the Gaussian model does not capture TCP retransmission, UDP fragmentation, or real network jitter.

Clearly documented as simulation. Presenter must note this to the audience.

R3

No real RTPS wire protocol — the system cannot interoperate with real DDS participants (Connext, CycloneDDS).

By design (ADR-001). Not a production system.

R4

matplotlib dark theme rendering differences — chart appearance may vary on different OS / display configurations.

Test with --generate-sample on the target presentation machine.

R5

_INSTALL_ERROR_PROB stochastic behavior — demo may occasionally show 0 errors (all ECUs succeed) or 2+ errors, making the demo unpredictable.

Presenter should run the demo 1-2 times in advance to pick a representative run for recording.

11.2. Technical Debts

Table 12. Technical debts
ID Debt

TD1

dds_abstraction.py does not implement on_deadline_missed / on_liveliness_changed callbacks as callable hooks; these are logged but not wired to application-level handlers in ecu.py or update_manager.py.

TD2

No unit tests are provided for dds_abstraction.py QoS behaviors (TRANSIENT_LOCAL replay, BEST_EFFORT drop rate distribution).

TD3

Multi-domain simulation (vehicle bus vs diagnostics domain) is described in README as an extension point but not implemented.

TD4

visualize_results.py has a hard dependency on pre-existing CSV schema; if metrics_collector.py schema changes, visualization breaks silently.

12. Glossary

Term Definition

AUTOSAR

AUTomotive Open System ARchitecture. A standardization consortium and software architecture framework for automotive ECUs. Includes Classic Platform (AUTOSAR CP) and Adaptive Platform (AUTOSAR AP).

ara::com

AUTOSAR Adaptive Platform communication API. Provides service-oriented communication; the DDS binding maps ara::com service interfaces to DDS topics and types.

CAN / CAN 2.0B

Controller Area Network. A legacy automotive bus protocol operating at up to 1 Mbps with 8-byte payload per frame and shared-bus arbitration. Referenced in this demo as the baseline for comparison against DDS.

CDR

Common Data Representation. The OMG standard serialization format used by RTPS/DDS for topic data. Produces compact binary payloads (~30 bytes for ECU state updates vs 50+ bytes for CAN multi-frame).

Deadline QoS

A DDS QoS policy requiring publishers to write at least once per specified period. Violations trigger on_requested_deadline_missed() callbacks on subscribers — used here as a distributed safety watchdog.

DDS

Data Distribution Service. An OMG standard for data-centric pub/sub middleware. Provides QoS-governed, topic-based communication with automatic discovery. The middleware paradigm simulated in this demo.

Domain

A DDS logical partition. Participants in the same domain can discover and communicate with each other. Participants in different domains are isolated. This demo uses domain_id=0.

Durability QoS

Governs whether data written before a subscriber joins is made available. TRANSIENT_LOCAL: writer caches last N samples for late joiners. VOLATILE: no cache; only future samples delivered.

ECU

Electronic Control Unit. An embedded computer in a vehicle that controls one or more subsystems (engine, brakes, OTA update client, etc.).

Jitter

max_latency - min_latency across all ECUs in one OTA cycle. Measures fleet synchronization spread. Lower jitter = more synchronized ECU updates.

Latency Budget QoS

A DDS QoS policy expressing the maximum acceptable end-to-end delivery delay. Used for monitoring and reporting; does not enforce the deadline at the transport layer.

Liveliness QoS

A DDS QoS policy requiring participants to periodically assert their liveness. If a participant fails to assert within lease_duration, other participants detect the liveliness loss. Used for ECU crash detection.

LIN

Local Interconnect Network. A low-cost, low-speed automotive serial bus (up to 20 kbps). Used for body electronics; even more constrained than CAN for OTA.

Multicast

Network delivery mode where a single packet is delivered to all group members. DDS uses UDP multicast for discovery and data delivery by default. A single DDS write() reaches all N subscribers simultaneously (O(1)).

OTA

Over-the-Air. Wireless delivery of software/firmware updates to deployed devices. In automotive, OTA updates are managed by the AUTOSAR UCM module.

PHM

Platform Health Management. AUTOSAR Adaptive module responsible for monitoring ECU health using supervision mechanisms (Alive, Deadline, Logical). Maps to DDS Deadline and Liveliness QoS in this demo.

pub/sub

Publish/Subscribe. A messaging pattern where producers (publishers) send data to named topics and consumers (subscribers) receive data from topics, decoupled in space and time.

QoS

Quality of Service. A set of configurable parameters governing data delivery behavior in DDS: reliability, durability, deadline, liveliness, latency budget, history, etc.

Reliability QoS

RELIABLE: ACK/NACK-based delivery; no message loss. BEST_EFFORT: no retransmission; ~3% drop tolerated for non-critical data.

RTPS

Real-Time Publish-Subscribe protocol. The OMG wire protocol underlying DDS. Includes Simple Discovery Protocol (SDP) for automatic participant and endpoint discovery over UDP.

SDP

Simple Discovery Protocol. The RTPS mechanism by which DDS participants automatically discover each other on the network. Eliminates the need for static configuration (cf. CAN DBC files).

SDV

Software-Defined Vehicle. Automotive architecture paradigm where vehicle functions are increasingly implemented in software rather than dedicated hardware, requiring scalable communication middleware like DDS.

TRANSIENT_LOCAL

DDS Durability QoS value. The writer caches the last history_depth samples. When a new subscriber joins, it automatically receives the cached samples — enabling late-joiner synchronization without re-broadcast.

UCM

Update and Configuration Management. AUTOSAR Adaptive module responsible for OTA firmware updates. Uses DDS (via ara::com) for coordination between the UCM Master and UCM clients on ECUs. This demo directly models UCM behavior.

Unicast

Network delivery mode where a packet is sent to a single recipient. CAN is effectively unicast per recipient — the UpdateManager must send N messages to reach N ECUs (O(N)).

VOLATILE

DDS Durability QoS value. No sample cache; subscribers only receive samples written after they joined. No late-joiner support.

14. Meta Discussions

14.1. Scope of this Arc42 Document

This arc42 document covers the simulation architecture of the Automotive DDS OTA Demo, not a production AUTOSAR Adaptive Platform deployment. The architectural decisions, constraints, and risks apply specifically to the Python simulation layer.

For a real AUTOSAR Adaptive Platform architecture, the arc42 views would additionally need to cover:

  • ara::com service interface definitions (.fidl / .fdepl)

  • ARXML element configuration (ServiceInstance, ProvidedServiceInstance, RequiredServiceInstance)

  • Execution Management (EM) manifests for process lifecycle

  • Communication Management (CM) DDS binding configuration

  • Platform Health Management (PHM) supervision configuration

14.2. Multi-domain Extension (Discussion)

The current demo uses a single DDS domain. A production SDV would typically use:

  • Domain 0: Vehicle bus (high-speed ECU-to-ECU, OTA commands)

  • Domain 1: Diagnostics (UDS over DDS, PHM health data)

  • Domain 2: Infotainment / HMI

A domain bridge (DDS Router) would route relevant topics between domains. This would add a new deployment view element and cross-cutting concept (domain routing policy).
