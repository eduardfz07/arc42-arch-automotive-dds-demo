Context and Scope

Business Context

The Automotive DDS OTA Demo operates within a DDS Domain 0 simulation. The external actors are:

Business context diagram
┌──────────────────────────────────────────────────────────────────────┐
│                        DDS Domain 0 (simulated)                       │
│                                                                        │
│  ┌──────────────────┐   OTAControl (RELIABLE, multicast)              │
│  │  UpdateManager   │──────────────────────────────────►  ECU_001..N  │
│  │  (UCM Master)    │◄──────────────────────────────────  ECU_001..N  │
│  │                  │   ECUStatus (RELIABLE + TRANSIENT_LOCAL)         │
│  └──────────────────┘                                                 │
│                                                                        │
│  ┌──────────────────┐                                                  │
│  │ MetricsCollector │◄── All state transitions (internal observer)    │
│  └──────────────────┘                                                  │
└──────────────────────────────────────────────────────────────────────┘
        │                               │
        ▼                               ▼
 [Presenter / User]            [CSV export / Matplotlib charts]
Table 1. Business context — external interfaces
Neighbour Description

Presenter / User

Invokes run_demo.py from the command line; configures number of ECUs, firmware version, QoS mode, and comparison flags.

CSV Data Store (data/)

MetricsCollector exports per-run metrics as timestamped CSV files for offline analysis.

Plot Store (plots/)

visualize_results.py writes 5 PNG charts; consumed by presentations / slides.

Sample Data Generator

generate_sample_data.py pre-populates data/ so visualization can run without a live demo.

Technical Context

The system is a single-process Python application; all participants share in-process message queues.

Table 2. Technical context
Interface Description

DDS Topics (in-process)

OTAControl (commands, RELIABLE QoS) and ECUStatus (state updates, RELIABLE + TRANSIENT_LOCAL). Implemented as Python dicts + threading queues in dds_abstraction.py.

CLI (argparse)

User configures --num-ecus, --firmware, --qos, --show-comparison, --duration flags.

Filesystem (CSV)

data/run_TIMESTAMP.csv — metrics output; read by visualize_results.py.

Filesystem (PNG)

plots/ — 5 chart PNGs generated by matplotlib.

stdout (ANSI)

Real-time ECU status table with colored state indicators during demo run.

Discovery: RTPS Simple Discovery Protocol is simulated by the participant registry inside dds_abstraction.py. No actual RTPS wire protocol is transmitted.

Transport: Intra-process Python threading.Queue objects replace UDP multicast sockets. The Gaussian latency model adds simulated network delay.
