=== Level 2 — Module Detail

==== `dds_abstraction.py` — DDS Simulation Layer

**Purpose:** Single-module replacement for a real DDS middleware stack. Simulates OMG DDS 1.4 pub/sub, QoS, and RTPS discovery within a single Python process.

**Key exported types and functions:**

[options="header",cols="1,3"]
|===
|Export|Description
| `ReliabilityKind` | Enum: `RELIABLE`, `BEST_EFFORT`
| `DurabilityKind` | Enum: `TRANSIENT_LOCAL`, `VOLATILE`
| `QoSProfile` | Dataclass: reliability, durability, history_depth, deadline_ms, latency_budget_ms, name
| `RELIABLE_QOS` | Pre-built QoS: RELIABLE + TRANSIENT_LOCAL, depth=10, deadline=100ms
| `BEST_EFFORT_QOS` | Pre-built QoS: BEST_EFFORT + VOLATILE, depth=1
| `CONTROL_QOS` | Pre-built QoS for OTA commands: RELIABLE + TRANSIENT_LOCAL, deadline=50ms
| `create_participant(domain_id)` | Registers a DDS participant in the simulated domain.
| `create_topic(name, type_name, qos)` | Registers a DDS topic in the participant registry.
| `create_writer(topic)` | Returns a writer handle for the given topic.
| `create_reader(topic, callback)` | Registers a callback invoked on each incoming sample; simulates `on_data_available`.
| `write(writer, data)` | Publishes a sample; applies simulated latency and BEST_EFFORT drop; dispatches to all matching readers.
| `shutdown(participant)` | Deregisters participant; triggers liveliness loss notifications.
|===

**Simulated QoS behaviors:**

* **RELIABLE:** All samples are delivered (no drop). ACK/NACK implied.
* **BEST_EFFORT:** ~3% random drop applied at `write()` time.
* **TRANSIENT_LOCAL:** Writer caches last `history_depth` samples; new readers receive the cache on discovery.
* **Deadline:** Tracked per-writer; `on_deadline_missed` callback fires if `write()` is not called within `deadline_ms`.
* **Liveliness:** Participants must assert liveness periodically; `on_liveliness_changed` fires on `shutdown()`.

==== `ecu.py` — Simulated ECU

**Purpose:** Models a single automotive ECU as a DDS participant. Runs its OTA state machine in a dedicated thread.

**State machine:**
....
IDLE ──► DOWNLOADING ──► VERIFYING ──► INSTALLING ──► REBOOTING ──► DONE
                                             │
                                             └──► ERROR (n% probability)
....

**DDS role:**

* **Subscriber** on `OTAControl` topic (RELIABLE + TRANSIENT_LOCAL) — receives `START_UPDATE` commands.
* **Publisher** on `ECUStatus` topic (RELIABLE + TRANSIENT_LOCAL) — publishes `ECUStateUpdate` samples at each state transition and during download progress (10 progress updates per stage).

**Key parameters:**

[options="header",cols="1,2"]
|===
|Parameter|Value
| `_INSTALL_ERROR_PROB` | 0.00 (0% installation failure rate, configurable)
| `ECUStatus` deadline | 100 ms
| `OTAControl` deadline | 50 ms
| Liveliness lease | 500 ms
|===

==== `update_manager.py` — UCM Master / OTA Coordinator

**Purpose:** Orchestrates the OTA update. Broadcasts the `START_UPDATE` DDS sample via multicast and monitors all ECUs until they reach `DONE` or `ERROR`.

**DDS role:**

* **Publisher** on `OTAControl` (CONTROL_QOS) — single `write()` reaches all N ECUs simultaneously.
* **Subscriber** on `ECUStatus` (RELIABLE_QOS) — receives state updates from every ECU; updates real-time ANSI status table.

**Key behaviors:**

* Displays a live ANSI-colored status table (ECU ID, state, progress %, latency, last update timestamp).
* Records per-ECU first-ACK latency (time from broadcast to first `ECUStatus` sample received).
* Detects `ERROR` state and reports resilience metrics.

==== `metrics_collector.py` — Metrics & Analysis Engine

**Purpose:** Records all state transitions, computes performance metrics, models CAN 2.0B overhead, and exports data.

**Sub-components:**

[options="header",cols="1,3"]
|===
|Class/Function|Description
| `MetricsCollector` | Records state transitions; computes per-ECU latency, jitter, throughput, end-to-end latency.
| `CANBusAnalyzer` | Models CAN 2.0B: frame count per update (ceil(payload/8)), bus load %, congestion at N ECUs.
| `generate_scalability_data()` | Produces a DataFrame for 5/10/20 ECU scenarios with DDS and CAN metrics side-by-side.
| `save_run_results()` | Exports per-run metrics to `data/run_TIMESTAMP.csv`.
|===

==== `run_demo.py` — Orchestrator

**Purpose:** Entry point. Parses CLI arguments, instantiates all components, runs the demo, and prints the metrics summary.

**CLI flags:**

[options="header",cols="1,2"]
|===
|Flag|Description
| `--num-ecus N` | Number of ECUs to simulate (default: 5)
| `--firmware X.Y.Z` | Firmware version string to broadcast (default: 2.0.0)
| `--qos {reliable,best_effort}` | QoS mode for OTA commands (default: reliable)
| `--show-comparison` | Run and display CAN vs DDS comparison table
| `--duration N` | Max demo duration in seconds (default: 30)
| `--inject-failure N` | ECU to inject the failure into (default: None)
| `--failure-at N` | Time after which the --inject-failure ECU shall fail
(default: 3)
| `--late-join-count N` | Number of ECUs that shall join the update process late
(default: 0)
| `--late-join-delay N` | Time after which the late joining ECUs shall join
(default: 3)
|===
